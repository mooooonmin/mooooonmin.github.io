## 1. 클라이언트-서버 구조
* **역할 분리:** 클라이언트는 요청(Request)을 보내고 대기함. 서버는 결과를 만들어서 응답(Response)함.
* 이렇게 분리되어 있어야 각각 독립적으로 진화가 가능함. (서버는 비즈니스 로직에만 집중, 클라이언트는 UI/UX에만 집중)

---

## 2. 무상태 프로토콜 (Stateless)
서버가 클라이언트의 상태를 보존하지 않는 특성임.

* **핵심 개념:** 서버는 "방금 요청한 놈이 아까 그놈인지" 모름.
* **인증 처리:** 서버가 기억을 못 하니까, 클라이언트가 매 요청마다 토큰(JWT)이나 세션 ID를 들고 가서 "나 아까 걔 맞음"이라고 입증해야 함.
* **장점:** * **무한 확장성:** 아무 서버나 응답해도 됨. 고객이 갑자기 늘어나면 서버를 한 대 더 늘리는(Scale-out) 게 매우 쉬움.
    * **서버 부하 감소:** 서버가 상태를 일일이 기억하고 저장할 필요가 없음.

---

## 3. Stateful vs Stateless 비교
상황에 따라 골라 써야 함.

| 구분 | Stateful (상태 유지) | Stateless (무상태) |
|:---|:---|:---|
| **연결성** | 지속적 연결 지향 (TCP/IP) | 비연결성 지향 (HTTP) |
| **특징** | 서버가 이전 상태를 다 기억함 | 서버는 이번 요청만 보고 처리함 |
| **장점** | 한 번 연결되면 통신이 효율적임 | 서버 확장성이 좋고 관리하기 편함 |
| **단점** | 서버가 터지면 상태가 다 날아감 | 매번 모든 정보를 다 보내야 함 |
| **예시** | 웹소켓 채팅, 온라인 게임 | 일반적인 웹 서비스, API |



---

## 4. 비연결성 (Connectionless)
* **원리:** 요청 주고 응답 받으면 바로 연결을 끊어버림.
* **왜 이럼?:** 수만 명이 동시에 접속해도 실제 동시에 요청을 보내는 건 소수임. 연결을 끊어두면 서버 자원을 효율적으로 아낄 수 있음.
* **한계:** 매번 새로 연결 맺는 시간(TCP 3-Way Handshake)이 걸림.
    * **해결:** 지금은 HTTP 지속 연결(Persistent Connections) 기술로 최적화해서 쓰고 있음.

> **💡 핵심 요약**
> * **클라이언트-서버:** 일 나누기.
> * **Stateless:** 기억 안 함 → 서버 늘리기 쉬움.
> * **비연결성:** 볼일 끝나면 끊음 → 서버 자원 아낌.
