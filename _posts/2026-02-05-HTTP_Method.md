## 1. 주요 HTTP Method

| 메서드 | 역할 | 멱등성 (Idempotent) | 주요 특징 |
| :---: | :---: | :---: | :--- |
| `GET` | **리소스 조회** | **O** | 서버의 상태를 변경하지 않으며, 주로 데이터를 받아올 때 사용. |
| `POST` | **요청 데이터 처리** | **X** | 주로 **리소스 등록**에 사용. 호출할 때마다 새로운 데이터가 생성될 수 있다. |
| `PUT` | **리소스 대체** | **O** | 리소스를 완전히 **덮어쓰기** 함. 리소스가 없으면 새로 생성. |
| `PATCH` | **리소스 부분 변경** | **△** | 리소스의 일부분만 수정. (설계 방식에 따라 멱등성이 깨질 수 있음) |
| `DELETE` | **리소스 삭제** | **O** | 특정 리소스를 삭제. 이미 삭제된 뒤 다시 호출해도 결과는 동일. |

## 2. HTTP 메서드의 멱등성 (Idempotency)

### 멱등성의 정의
> **"같은 요청을 여러 번 호출해도 결과가 항상 같고, 서버의 데이터 상태가 첫 번째 호출 이후와 동일하게 유지되는 특성"**

* **판단 기준:** 외부 요인(DB 강제 삭제 등)으로 인한 변경은 고려하지 않으며, 오직 **해당 API의 호출 결과**가 동일한지를 확인.
* **`POST`가 멱등하지 않은 이유:** 게시글 쓰기 버튼을 5번 누르면 게시글이 5개 생기는 것처럼, 호출 시마다 서버 상태가 변하기 때문.

### 재요청(Retry) 판단 근거
네트워크 장애나 타임아웃 등으로 응답을 받지 못했을 때, 클라이언트가 **"다시 요청해도 안전한가?"**를 결정하는 기준.

* **멱등한 메서드 (`GET`, `PUT`, `DELETE`):** 정상 응답을 못 받았다면 안심하고 다시 요청을 보낼 수 있다.
* **멱등하지 않은 메서드 (`POST`):** 중복 결제, 중복 주문 등의 위험이 있으므로 단순 재요청은 지양해야 하며, 별도의 처리가 필요.

## 3. 안전(Safe)과 멱등(Idempotent)의 차이

| 개념 | 의미 | 해당 메서드 |
| :--- | :--- | :--- |
| **안전 (Safe)** | 호출해도 리소스가 수정/삭제되지 않음 | `GET` |
| **멱등 (Idempotent)** | 여러 번 호출해도 결과가 동일함 | `GET`, `PUT`, `DELETE` |

> **💡 핵심 요약**
> * **안전**은 리소스의 **변형**이 없는가에 집중하고, **멱등**은 여러 번 수행 시 **결과**가 같은가에 집중.
> * 따라서 `PUT`이나 `DELETE`는 리소스를 변경하므로 **안전하지 않지만**, 여러 번 수행해도 상태가 유지되므로 **멱등**함.
