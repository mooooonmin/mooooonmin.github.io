## 1. EA(Enterprise Architecture)
* 기업의 목표와 요구를 잘 지원하기 위해 IT 인프라의 각 부분들이 어떻게 구성되고 작동되어야 하는가를 체계적으로 기술하는 것. 
* 실제 목적은 기업의 특성이나 기업의 상황에 따라 다를 수 있음.

## 2. 아키텍처의 3가지 구성요소
* **모델**
* **규칙**
* **계획**

## 3. 전사의 범위
* 기업의 규모와 구조에 따라 달라짐.
* 대규모의 기업은 여러 개의 전사로 구성될 수 있음.
* EA 수립 목적에 따라 그 범위를 기업 전체로 할 수도 있고 특정 전사로 제한할 수도 있음.
* 전사는 기업이나 기관과 동일한 범위가 아닐 수 있음.

## 4. 전사 아키텍처 프레임워크(EA 프레임워크)
전사 아키텍처 활동 산출물을 분류하고 조직화하고 유지하기 위한 툴.

| 프레임워크 | 설명 |
|---|---|
| **자크만 프레임워크(ZEAF)** | 다섯 가지 관점(planner, owner, designer, builder, sub-contractor)과 여섯 가지 묘사 방법(data, function, network, people, time, motivation). |
| **미연방정부 프레임워크(FEAF)** | 참조모델(BRM, DRM, SRM, TRM, PRM) 기반의 EA 프레임워크. |
| **미 재무성 EAF(TEAF)** | How-Where-When을 표현하는 기능, What-How much-How Freq를 표현하는 정보, Who-Why를 표현하는 조직, Enable를 표현하는 인프라 관점 중시 / 계획자, 설계자, 소유자, 개발자 등의 관점으로 구분한 아키텍처 매트릭스. |
| **미 국방성 아키텍처 프레임워크(DoDAF)** | 효과적인 작전 수행을 위해 무기 체계 간 상호운용성 보장을 위해 도입. |
| **오픈 그룹의 EA 프레임워크(TOGAF)** | 각종 참조 모델의 활용 관계를 잘 정의함. 메타 모델에 근거한 연속체 개념을 도입하여 아키텍처 파악함으로써 조직의 특수성이 프레임워크에 반영되기 어려움. |

## 5. EA 프레임워크 구성요소

| 구분 | 구성 요소 | 설명 |
|:---:|:---|:---|
| **EA 정책** | **아키텍처 매트릭스** | 전사아키텍처의 정보를 체계적으로 분류한 틀. 기업이 관리하려고 하는 전사 아키텍처 정보의 수준과 활용 계층을 결정. |
| | **EA 비전** | 전사아키텍처 구축 목표와 목표를 효과적으로 달성하기 위한 전략 및 방향을 포함한 미래상. |
| | **EA 원칙** | 전사아키텍처 정보를 효율적으로 구축하고 기업 목적에 맞게 전사아키텍처를 효과적으로 활용하기 위해 조직 구성원이 공유해야 할 규범. |
| **EA 정보** | **현행 아키텍처** | 각 아키텍처 도메인별 기업의 현재의 상태를 정의한 것. |
| | **목표 아키텍처** | 각 아키텍처 도메인별 기업이 궁극적으로 달성하고자 하는 목표 아키텍처 상태를 정의한 것. |
| | **전사아키텍처 이행계획** | 각 아키텍처 도메인별로 현재 모습에서 바람직한 목표 모습으로 이행하기 위한 이행전략과 이행계획을 정의한 것. |
| **EA 관리** | **전사아키텍처 관리체계**<br>(거버넌스) | 구축된 전사아키텍처를 유지하고 개선하기 위한 제도적 기반을 수립하는 것 / BA, AA, DA, TA의 유기적 연결. |
| | **전사아키텍처 관리시스템** | 전사아키텍처 정보관리의 효율성을 제고하고, 전사아키텍처 정보의 공유를 활성화하기 위해 구축하는 정보시스템. |
| | **전사아키텍처 평가** | 전사아키텍처 관리와 활용 수준의 제고를 위해서는 전사아키텍처에 대한 주기적인 평가 및 전사아키텍처 성숙모형이 필요. |

## 6. 아키텍처 매트릭스 정의 시 고려 사항
* 조직 내 다양한 계층의 사람들이 매트릭스에 포함된 산출물이 범위와 목적에 적합하게 정의되었음을 확신할 수 있어야 함.
* 아키텍처 매트릭스는 IT 조직의 성숙도를 고려해 정의.
* 전사아키텍처 정보를 공유정보로 구축하기 위해서는 EA 산출물에 포함된 정보를 중복이 없고 상호관계가 유기적으로 연결될 수 있는 ‘EA 정보 구성 요소’를 추가로 정의해야 함.
* 기업의 조직문화, 의사결정 등에 따라 산출물이 달라지므로 기업의 조직문화와 의사결정구조를 고려해 정의.
* 실제 시스템과 아키텍처 개발표준에 대한 준수성을 높이고 조직별로 통일된 접근이 가능하도록 정의.

## 7. 참조모델
* 정부, 산업, 기업군 단위로 아키텍처 구성요소를 식별하여 표준화한 것. 
* 기업이나 기업의 EA 수립에 참조하는 모델. (업무, 서비스, 기술, 데이터 및 성과 참조 모델)
* 상세 정의/구축방법/사례/활용은 `27~31` 절에서 통합 설명.

## 8. 아키텍처 도메인 구성
* 아키텍처 매트릭스 상에서 뷰의 관점으로 아키텍처 영역을 구분한 것.
* 기업이나 조직이 전사아키텍처 정보를 체계적으로 분류한 아키텍처 매트릭스를 어떻게 정의하느냐에 따라 도메인 구성은 달라짐.

| 도메인 | 핵심 정의 |
|---|---|
| **비즈니스 아키텍처** | 기업의 경영 전략 및 비전 모델을 기반으로 비즈니스 수행을 위한 조직·업무 프로세스를 정의. |
| **애플리케이션 아키텍처** | 비즈니스 수행을 위해 필요한 애플리케이션을 어떻게 정의하고 배치할지 정의. |
| **데이터 아키텍처** | 비즈니스 수행에 필요한 데이터를 어떻게 정의하고 유지할지 정의. |
| **기술 아키텍처** | 애플리케이션·데이터를 실행하기 위한 기술 인프라를 어떻게 구성하고 관리할지 정의. |

### EA 정보(현행/목표/이행) 요약
* **현행 아키텍처(As-Is)**: 기업의 현재 상태.
* **목표 아키텍처(To-Be)**: 기업이 궁극적으로 달성하고자 하는 목표 상태.
* **전사아키텍처 이행계획**: 현재 모습에서 바람직한 목표 모습으로 이행하기 위한 이행 전략 및 이행 계획.

## 9. 데이터 아키텍처
* 기업이나 조직의 업무 수행에 필요한 데이터의 구조를 체계적으로 정의하여 구축하고 관리하기 위한 방법.
* 데이터가 진정한 자산으로서 가치를 가지려면 체계적으로 잘 정의되고 우수한 품질이 유지되어야 함.

## 10. 데이터아키텍처 구성

| 구성 요소 | 관점 | 설명 |
|---|:---:|---|
| **전사 데이터 영역 모델** | 계획자 관점 | 상위 수준의 전사 데이터 영역 분류(주제 영역). 개괄 데이터 모델. 전사 범위/규모에 따라 계층적 구성. |
| **개념 데이터 모델** | 책임자 관점 | 단위 주제 영역 또는 핵심 엔터티 정도를 표현. 윤곽/구도 파악 용도. 필수 구성 대상으로 보거나 생략 가능. |
| **논리 데이터 모델** | 설계자 관점 | 상세 구조 구체화. 식별자, 속성, 관계, 서브타입 정의. 업무 규칙 명확화 및 무결성 보장. 전사 표준 관리 필수. |
| **물리 데이터 모델** | 개발자 관점 | 특정 DBMS나 분산 서버 환경에 적합한 구조 설계. PK 컬럼 순서 조정, 인덱스 설계, 비정규화, 파티셔닝 등 성능 고려. |

## 13. 현행 아키텍처 정보 구축
* 상위 수준의 분류 체계 정의 후 나머지 하위 정보 구축은 병렬적으로 수행하는 것이 효율적임.
* 현행 애플리케이션 아키텍처는 업무 기능 분류 기반의 전사 앱 분류와 연관관계 분석부터 시작.

| 방법 | 설명 |
|:---:|---|
| **하향식 방법** | 필요한 데이터 요건 식별하여 모델 표현. |
| **상향식 방법** | 시스템 데이터 정보를 분석하여 정련하여 표현. |

## 14. 목표 아키텍처 정보 구축
* 현행의 문제점과 개선사항 도출하여 반영.
* **비즈니스 아키텍처(BA)를 먼저 정의**하고 이를 지원하는 데이터, 애플리케이션, 기술 아키텍처를 정의하는 것이 바람직.
* 초기에는 개념적 수준까지 정의하는 것이 일반적.

## 15. EA 관리

| 구분 | 설명 |
|---|---|
| **전사아키텍처 관리 체계(거버넌스)** | 구축된 전사아키텍처를 유지·개선하기 위한 제도적 기반으로, 정의된 전사아키텍처 원칙을 준수·통제하기 위한 조직과 프로세스를 정의한 것. |
| **전사아키텍처 관리 시스템** | 전사아키텍처 정보관리 효율성을 높이고 정보 공유를 활성화하기 위해 구축하는 정보시스템. 일반적으로 모델링 도구, 전사아키텍처 리포지터리, 전사아키텍처 포털로 구성. |
| **전사아키텍처 평가** | 전사아키텍처 구축·활용 수준을 점검하고 개선점을 도출하기 위한 주기적 평가 활동. 보통 **전사아키텍처 성숙모형**을 기준으로 수행. |

## 19. 범정부 EA 핵심 모델 및 지원시스템

| 구분 | 설명 |
|---|---|
| **범정부 전사아키텍처 참조모델** | 아키텍처 구성 요소 표준화 및 분류를 위한 참조 기준. BRM(업무기능), SRM(서비스컴포넌트), DRM(데이터), TRM(기술), PRM(성과)로 구성되며, 기관은 범정부 참조모델을 활용하거나 자체 참조모델 수립 시 준거로 사용. |
| **범정부 전사아키텍처 산출물 메타모델** | 아키텍처를 구성하는 정보와 정보 간 관계를 정의하는 모델(산출물 종류/구성). 전사아키텍처는 필수 산출물과 보조 산출물로 구성되며, 각 기관은 필수 산출물 및 필수 요소를 반드시 정의해야 함. |
| **범정부 전사아키텍처 성숙도모델** | 아키텍처 수준과 향후 발전 방향 수립을 위한 모델. 일반적으로 1~5단계 성숙도 수준과 측정 항목으로 구성하며, 매년 전사아키텍처 추진 실태와 함께 성숙도를 측정·관리. |
| **범정부 전사아키텍처 지원시스템(GITAMS)** | 아키텍처 관련 표준/기준 제공, 기관 간 아키텍처 정보 공유·활용을 지원하는 시스템. 각 기관은 필수 산출물에 해당하는 정보를 매년 GITAMS에 등록해야 함. |

## 20. 전사아키텍처 관련 데이터아키텍처 전문가 역할

## 21. 전사아키텍처 정책

## 22. 비즈니스 아키텍처
* 기업의 경영목표 달성을 위한 업무 구조를 정의하는 아키텍처 영역으로, 기업의 업무와 서비스를 명확히 하는 데 목적이 있음.

### 22.1 전사 사업 모델(계획자 관점)
* 전사의 범위 정의부터 시작하며, 전사를 둘러싼 내·외부 이해관계자와 외부 개체의 가치사슬을 분석해 전사를 정의함.
* 전사 가치사슬(Value Chain) 관점에서 핵심 활동과 지원 활동을 구조화해 식별함.

### 22.2 조직 모델(계획자 관점)
* 비즈니스를 수행하는 지리적 위치와 내부 개체를 도출해 기업의 조직 구조와 업무 분석 틀을 정의함.

### 22.3 업무기능 모델(책임자 관점)
* 기업의 업무 기능을 계층적으로 분할하고 기능 내용을 정의함.
* 업무기능은 유사성과 연관성 기준으로 정의하고, 상위 기능은 하위 기능의 합으로 완전하게 설명 가능해야 함.

### 22.4 프로세스 모델(설계자 관점)
* 업무기능을 상세화해 계층적으로 프로세스를 분할하고 각 프로세스의 활동 내용을 정의함.

### 22.5 업무 매뉴얼(개발자 관점)
* 실제 업무 수행을 위한 구체 절차, 양식, 역할 및 책임을 문서화해 실행 가능한 수준으로 정리함.

## 23. 애플리케이션 아키텍처
* 비즈니스 수행을 지원하는 응용시스템의 구조와 관계를 정의하는 영역.

### 23.1 전사애플리케이션 영역 모델(계획자 관점)
* 전사 단위 응용시스템을 영역별로 분류하고, 전반적 구성과 경계를 정의함.

### 23.2 애플리케이션 모델(책임자 관점)
* 주요 애플리케이션의 기능 구성과 시스템 간 연관관계를 정의함.

### 23.3 컴포넌트 모델/클래스 모델(설계자 관점)
* 애플리케이션 내부 구성요소를 모듈 수준으로 분해해 구조와 상호작용을 정의함.

### 23.4 프로그램 목록(개발자 관점)
* 구현 단위 프로그램을 목록화하고 배치/운영 관점에서 관리함.

## 24. 데이터 아키텍처
* 데이터 구조를 체계적으로 정의하는 것.
* 관리 대상은 업무데이터와 메타데이터를 모두 포함함.

### 24.1 전사 데이터 영역 모델(계획자 관점)
* 개괄 데이터 모델로, 상위 수준의 전사 데이터 영역(주제영역) 분류와 구성도를 정의함.

### 24.2 개념 데이터 모델(책임자 관점)
* 전사 수준 데이터 모델 또는 단위 주제영역별 핵심 엔터티 중심으로 표현하는 모델.
* 전사 데이터 구조를 전체적으로 파악하고 데이터 아키텍처를 관리하는 기본 틀로 활용됨.

### 24.3 논리 데이터 모델(설계자 관점)
* 개념 모델의 주제영역/핵심 엔터티를 바탕으로 업무 요건을 충족하는 데이터 상세 구조를 논리적으로 구체화함.
* 엔터티, 속성의 명칭·정의·규칙·코드 등을 전사 차원의 표준으로 정의하고 관리해야 함.

### 24.4 물리 데이터 모델(개발자 관점)
* 기술 환경과 성능을 고려해 물리 데이터 구조를 설계하고 데이터베이스 객체를 정의함.
* 데이터 무결성을 보완하고, 데이터 분산 설계에 따른 추가 무결성 규칙을 정의함.

## 25. 기술 아키텍처
* 비즈니스·애플리케이션·데이터 아키텍처를 안정적으로 실행하기 위한 기술 인프라 구조와 표준을 정의하는 영역.

### 25.1 전사기술영역 모델/기술 참조 모델(계획자 관점)
* 전사 차원의 기술 영역을 분류하고 표준 기술 체계를 제시함.
* 공통 서비스, 플랫폼 서비스, 네트워크 서비스, 보안 서비스 등 기술 구성요소의 참조 구조를 정의함.

### 25.2 표준 프로파일(책임자 관점)
* 기술 참조 모델에 명시된 서비스를 지원하기 위한 정보기술 표준의 집합.
* 시스템의 이식성, 확장성, 상호운용성, 호환성 향상을 목표로 함.

### 25.3 기술 아키텍처 모델(설계자 관점)
* 서비스 카테고리별 아키텍처 패턴을 정의하거나, 기업의 소프트웨어·하드웨어·네트워크 구성요소에 대한 배치 구조를 정의함.

### 25.4 기술자원 목록/제품 목록(개발자 관점)
* 실제 도입·운영 대상 기술자원과 제품 정보를 목록화하여 관리함.

## 27. 참조모델 정의 및 TRM 비교

### 27.1 참조모델 개념
* 아키텍처 구성요소를 식별하고 표준화해 기관/기업이 전사아키텍처를 수립할 때 참조하는 추상화된 모델.

### 27.2 참조모델 현황(핵심)
* **성과 참조모델(PRM)**: 정보화 성과 측정을 위한 항목·지표·방법을 제시.
* **업무 참조모델(BRM)**: 업무 아키텍처의 기준이며, 대상 기관의 사업/업무를 전사 관점에서 분류·정의.
* **서비스 참조모델(SRM)**: 응용 아키텍처 기준으로, 응용 서비스 기능을 분류·정의.
* **데이터 참조모델(DRM)**: 데이터 아키텍처 기준으로, 기관 간 교환되는 주요 데이터 요소를 식별·정의·표준화.
* **기술 참조모델(TRM)**: 기술 아키텍처의 기준으로, 정보기술을 분류·식별.

### 27.3 TRM 관점별 특징 비교
| 모델 | 특징 |
|---|---|
| **미연방 TRM** | 업무·서비스·데이터와의 연계성이 높고, 내/외부 연계 구조 설명이 용이함. 다만 대내 IT 서비스 참조모델로 직접 적용 시 환경 차이에 따른 보완이 필요. |
| **오픈그룹 TRM** | 애플리케이션 이식성과 상호운용성 확보를 위한 기반 플랫폼·인터페이스를 중시. 표준 중심 구조로 플랫폼/네트워크/통신 서비스 분류에 강점. |
| **가트너 TRM** | 데스크톱 애플리케이션과 엔터프라이즈 애플리케이션을 구분해 실용적 분류 체계를 제공. 다만 상세 분류 기준은 조직별로 추가 정의가 필요. |
| **정보화진흥원 TRM** | 업무/서비스/데이터/네트워크와의 연계성 관점이 강하고, 국내 정보화 환경을 고려한 기술 선택에 유리. |

## 28. 참조모델 구축방법 및 사례

### 28.1 참조모델 구축방법
* 공통적인 특성을 추출해 산업군에 맞게 범용적으로 재사용 가능한 참조모델로 구축.
* 복잡하고 대표성 있는 기업을 선정해 해당 기업의 아키텍처를 표준으로 삼고, 유사 기업에 재활용 가능한 형태로 구축.

### 28.2 업무참조모델(BRM) 사례
* 특정 기관의 독립적인 업무 기능을 중심으로 정의되는 참조모델.
* 일반적으로 기능 분류(대기능/중기능/소기능)와 과제 분류(비전/정책목표/이행과제/관리과제)를 연결해 구조화함.
* 단위 과제 수준에서는 업무 처리 절차 흐름, 유관 정보, 속성 정보, 연계 과제 정보를 함께 관리해 업무-데이터 연계를 강화함.

---

## 29. 데이터참조모델(DRM) 구조와 활용
* 기관 간 공통 정보의 파악·공유·재사용을 지원하기 위한 모델.
* 범정부 데이터 분류와 데이터 표준화·관리 체계를 통해 기관 간 상호운용성을 높이는 것을 목표로 함.

### 29.1 범정부 데이터 모델
* 범정부에서 구축·활용 중인 데이터를 식별하고, 데이터 영역 간 관계를 체계적으로 도식화한 모델.

### 29.2 DRM 핵심 구성요소
| 구성요소 | 설명 |
|---|---|
| **데이터 분류** | 데이터 분류 기준을 정의해 분류체계 기반의 데이터 관리·검색이 가능하도록 함. |
| **데이터 구조** | 범위 내 모든 데이터 요소와 소유자, 표준 항목 정의, 요소 간 관계 정보를 저장·관리. |
| **데이터 교환** | 교환 대상 정의, 메시지 구조/방식 표준, 교환 이력 관리 기준을 정의. |
| **데이터 관리** | 데이터 품질·표준화·보안 유지를 위한 정책/규칙/프로세스/조직 구조를 정의. |

### 29.3 기대 효과
* 기관 간 정보 공유 기반 강화.
* 데이터 표준화 및 재사용 촉진.
* 데이터 교환 구조 재사용을 통한 상호운용성 향상.

## 30. 성과참조모델(PRM)
* 정보화 성과관리를 위해 구성요소와 관계를 정의하는 모델로, 정보화 성과 제고와 사업 품질 향상을 위한 기준을 제공함.
* 정보화 사업의 입력, 과정, 결과, 최종 성과 간 인과관계를 명확히 해 계층별 성과 연계를 강화함.

### 30.1 PRM 구조(핵심 관점)
| 계층 | 주요 내용 |
|---|---|
| **정책 목표/미션·업무** | 기관의 미션과 전략목표를 성과관리의 최상위 기준으로 설정. |
| **고객 관점** | 고객 만족도, 서비스 수준, 서비스 범위 등 대국민 성과 지표와 연결. |
| **프로세스 및 활동** | 재무, 생산성, 업무품질, 보안 등 내부 프로세스 성과를 측정·개선. |
| **자원 관점** | 인적자원, 정보기술, 연관자원(관리체계/조직문화 포함) 기반의 실행역량을 관리. |

### 30.2 활용 포인트
* 정책목표와 실행성과를 연계해 성과관리의 일관성을 확보.
* 고객 가치, 내부 프로세스, 자원 역량을 함께 점검해 균형 있는 성과평가 가능.

## 31. 참조모델 활용

| 모델 | 활용방안 | 기대효과 |
|---|---|---|
| **업무 참조모델(BRM)** | 개선 대상 업무를 업무 참조모델 기준으로 파악하고, 기관 비즈니스 아키텍처 정의 시 참조 | 기관 간 업무 흐름 촉진, 업무 개선 기준 제공, 업무 생산성 제고, 성과 측정 용이 |
| **데이터 참조모델(DRM)** | 개선 대상 데이터 구조를 DRM 기준으로 파악하고, 기관 데이터 아키텍처 정의 시 참조 | 데이터 상호운용성 및 교환 촉진, 데이터 표준화·재사용 확대, 중복 배제, 통합 데이터 활용 |
| **서비스 참조모델(SRM)** | 개선 대상 애플리케이션 서비스를 SRM 기준으로 파악하고, 기관 애플리케이션 아키텍처 정의 시 참조 | 시스템 간 상호운용성 향상, 신뢰성 있는 시스템 구축, 변화 대응력 향상, 개발 생산성·품질 개선 |
| **기술 참조모델(TRM)** | 개선 대상 기술 인프라를 TRM 기준으로 파악하고, 기관 기술 아키텍처 정의 시 참조 | 시스템 간 상호운용성/이식성/확장성 향상, 벤더 독립성 확보, 재활용·리소스 공유 촉진 |
| **성과 참조모델(PRM)** | 평가 체계와 지표 항목을 PRM 기준으로 정렬해 기관 성과평가 체계를 정의 | 체계적 성과 측정·관리, 성과기반 예산 편성 지원, 중복 투자 최소화, IT 자원 효율 관리 |

## 32. 전사아키텍처 프로세스 개요 및 구성

### 32.1 전사아키텍처 프로세스 개요
* 전사아키텍처를 구축하고 관리하는 전체 절차로, 작업 단계·공정·작업 내용을 정의한 것.
* 전사아키텍처 프레임워크 내 각 구성요소를 정의하기 위한 활동을 포함함.

### 32.2 전사아키텍처 프로세스 구성(단계-공정-내용)
| 단계 | 공정 | 내용 |
|---|---|---|
| **EA 비전 수립** | EA 방향 수립 | 내·외부 EA 환경 분석, 기업의 EA 목적/방향 정의, 기업의 EA 프레임워크 정의 |
| **EA 구축** | EA 정보 구성 정의 | 아키텍처 매트릭스 정의, EA 정보 구성 요소 정의, EA 참조모델 정의, EA 원칙 수립 |
| **EA 구축** | EA 정보 구축 | EA 자료 수집, 현행 아키텍처 정보 구축, 목표 아키텍처 정보 구축 |
| **EA 관리 정의** | EA 관리 체계 구축 | EA 정보 운영·활용을 위한 조직/프로세스 정의, 거버넌스 활동 및 내부 통제 체계 기반 교육 수행 |
| **EA 관리 정의** | EA 관리 시스템 구축 | EA 정보를 관리하기 위한 도구 선정, EA 정보를 관리하기 위한 시스템 구축 |
| **EA 활용 정의** | EA 이행 계획 | 목표 아키텍처를 달성하기 위한 중장기 이행 계획 수립 |
| **EA 활용 정의** | EA 정보 활용 | EA 정보를 적용한 IT 관련 업무 수행(IT 기획/구축/통제) |
