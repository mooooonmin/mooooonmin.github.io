## 1. GET vs POST 한눈에 비교
핵심은 '용도'와 '데이터 전달 방식'의 차이임.

| 구분 | GET | POST |
| :--- | :--- | :--- |
| **주요 역할** | 리소스 **조회** (Read) | 데이터 **처리/등록** (Create) |
| **데이터 위치** | URL의 Query String | 메시지 바디 (Body) |
| **캐싱 여부** | 가능함 (브라우저가 결과 저장함) | 불가능함 |
| **브라우저 기록** | 히스토리에 남음 | 남지 않음 |
| **보안성** | 데이터가 URL에 다 노출됨 (낮음) | 바디에 숨겨서 보냄 (비교적 높음) |



---

## 2. 메서드별 상세 특징

### 🔍 GET: 리소스 조회
* 클라이언트가 서버에 정보를 요청할 때 씀.
* URL 끝에 `?key=value` 형태의 **Query String**을 붙여서 보냄.
* 한 번 요청한 결과는 브라우저가 캐싱해 두기 때문에, 똑같은 요청을 하면 훨씬 빠르게 결과를 가져올 수 있음.

### ✉️ POST: 데이터 처리 및 생성
* 서버가 데이터를 처리하도록 요청할 때 씀. 주로 새로운 리소스를 등록할 때 사용함.
* 데이터가 Body에 담겨 가기 때문에 GET보다 대용량 데이터를 보내기 좋음.
* 단순히 등록뿐만 아니라, 특정 프로세스를 실행하거나 데이터를 변경할 때도 광범위하게 쓰임.

---

## 3. PUT vs PATCH: 리소스 수정
둘 다 수정을 위한 메서드지만, 수정하는 '범위'가 다름.

### 🔄 PUT (리소스 전체 대체)
* 대상 리소스를 **통째로** 갈아치움.
* 보낼 때 수정 안 할 필드까지 다 보내야 함. 안 보내면 해당 필드가 누락되거나 기본값으로 덮어씌워질 수 있음.
* 리소스가 없으면 새로 생성함.

### 🩹 PATCH (리소스 부분 수정)
* 리소스의 **일부분만** 바꿀 때 사용함.
* 딱 바꾸고 싶은 데이터만 보내면 됨. 기존의 다른 필드들은 그대로 유지됨.

## 4. 코드 예시로 보는 차이 (PUT vs PATCH)

**기본 데이터 (User/10)**
```json
{
  "name": "Noname",
  "language": "JAVA"
}
```



### ① PUT 요청 시 (name만 보내면)

**요청 데이터:**
```json
{
  "name": "TEST"
}
```

**결과 데이터:**
```json
{
  "name": "TEST"
}
```

**설명:** 기존에 있던 `language` 필드가 사라짐. 리소스를 **통째로 대체**하기 때문에 요청 시 보내지 않은 데이터는 삭제되거나 기본값으로 덮어씌워짐.

---

### ② PATCH 요청 시 (name만 보내면)

**요청 데이터:**
```json
{
  "name": "TEST"
}
```

**결과 데이터:**
```json
{
  "name": "TEST",
  "language": "JAVA"
}
```

**설명:** 기존 `language` 필드가 그대로 유지됨. 변경을 요청한 `name` 필드만 **부분적으로 수정**하기 때문임.

## 💡 추가 내용용

**"GET은 캐싱이 되나요?"**
* ✅ GET은 리소스 조회 목적이라 브라우저가 결과를 기억해 둠. 똑같은 요청을 또 하면 서버까지 안 가고 캐시에서 꺼내 써서 훨씬 빠름.

**"PUT이랑 PATCH 중 뭐가 더 안전함?"**
* 단순히 안전의 문제는 아니고 상황에 맞게 써야 함. 보통 리소스 전체를 갈아치울 땐 **PUT**, 일부 필드만 바꿀 땐 **PATCH**가 정석임. (참고로 PUT은 멱등하지만, PATCH는 설계에 따라 멱등하지 않을 수도 있음.)

**"보안 데이터는 어떻게 보냄?"**
* 비밀번호나 개인정보 같은 건 무조건 **POST** Body에 담아서 보내야 함. GET은 URL 쿼리 스트링에 데이터가 다 노출되고 브라우저 기록에도 남아서 매우 위험함!
