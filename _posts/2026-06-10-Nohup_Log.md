## 1. 백그라운드 프로세스 로그 확인이 필요한 이유

`nohup`과 `&`를 사용하면 Spring Boot 서버를 백그라운드에서 실행할 수 있다.

```bash
nohup java -jar linux-springboot-0.0.1-SNAPSHOT.jar &
```

백그라운드로 실행하면 같은 터미널에서 다른 명령어를 입력할 수 있다는 장점이 있다.
하지만 포그라운드 실행과 달리 서버 로그가 터미널에 실시간으로 보이지 않는다.

Spring Boot 서버가 정상적으로 실행됐는지, 중간에 에러가 발생했는지 확인하려면 백그라운드에서 실행 중인 프로세스의 로그를 확인할 수 있어야 한다.

---

## 2. 기존 Spring Boot 프로세스 종료하기

먼저 실행 중인 Spring Boot 프로세스가 있다면 종료한다.

```bash
ps aux | grep java
kill [PID 값]
```

`ps aux | grep java`로 Java 프로세스를 찾고, 종료할 프로세스의 PID 값을 `kill` 명령어에 전달한다.

```bash
kill [PID 값]
```

프로세스가 종료됐는지 다시 확인한다.

```bash
ps aux | grep java
```

---

## 3. 포그라운드 실행에서는 로그가 바로 보인다

Spring Boot 서버를 포그라운드에서 실행해보자.

```bash
cd ~/linux-springboot/build/libs
java -jar linux-springboot-0.0.1-SNAPSHOT.jar
```

포그라운드에서 실행하면 Spring Boot가 출력하는 로그가 터미널 화면에 바로 표시된다.
이 로그를 보면 서버가 정상적으로 실행됐는지, 어떤 에러가 발생했는지 간접적으로 확인할 수 있다.

다음 실습을 위해 실행 중인 서버를 종료한다.

```text
Ctrl + c
```

---

## 4. 백그라운드 실행에서는 로그가 터미널에 바로 보이지 않는다

이번에는 Spring Boot 서버를 백그라운드에서 실행한다.

```bash
nohup java -jar linux-springboot-0.0.1-SNAPSHOT.jar &
```

백그라운드에서 실행하면 다시 명령어를 입력할 수 있는 상태가 된다.
대신 포그라운드 실행처럼 Spring Boot 로그가 터미널에 계속 표시되지는 않는다.

그렇다면 로그는 사라진 것일까?
그렇지 않다.

GNU Coreutils 문서에 따르면 `nohup`은 표준 출력이 터미널인 경우 출력 내용을 `nohup.out` 파일에 이어서 저장한다. [1]

명령어를 실행했을 때 아래와 비슷한 메시지가 보일 수 있다.

```text
appending output to 'nohup.out'
```

이 메시지는 프로세스의 출력값을 `nohup.out` 파일에 추가해서 저장하겠다는 의미다.

---

## 5. nohup.out 파일 확인하기

확실한 테스트를 위해 기존 Spring Boot 프로세스와 기존 `nohup.out` 파일을 정리한 뒤 다시 실행해보자.

```bash
ps aux | grep java
kill [PID 값]
```

기존 `nohup.out` 파일이 있다면 삭제한다.

```bash
ls
rm -rf nohup.out
ls
```

다시 Spring Boot 서버를 백그라운드에서 실행한다.

```bash
nohup java -jar linux-springboot-0.0.1-SNAPSHOT.jar &
```

`nohup.out` 파일이 생성됐는지 확인한다.

```bash
ls
```

로그 내용을 확인한다.

```bash
cat nohup.out
```

`nohup.out` 파일 안에 Spring Boot 로그가 기록되어 있다면 정상이다.

## 6. nohup.out이 아닌 다른 파일에 로그 남기기

기본 로그 파일인 `nohup.out` 대신 직접 정한 파일에 로그를 남길 수도 있다.
이때는 리다이렉션을 사용한다.

```bash
rm -rf nohup.out
nohup java -jar linux-springboot-0.0.1-SNAPSHOT.jar >> result.log 2>&1 &
```

로그 파일을 확인한다.

```bash
ls
cat result.log
```

위 명령어는 Spring Boot 프로세스에서 발생하는 표준 출력과 표준 에러 출력을 `result.log` 파일에 저장한다.

| 표현 | 의미 |
|---|---|
| `>> result.log` | 표준 출력을 `result.log` 파일 끝에 이어서 저장한다. |
| `2>&1` | 표준 에러 출력을 표준 출력과 같은 곳으로 보낸다. |
| `&` | 명령어를 백그라운드에서 실행한다. |

Bash 문서에 따르면 `>>`는 출력 내용을 파일 끝에 추가하고, `>>word 2>&1`은 표준 출력과 표준 에러 출력을 같은 파일에 이어서 저장하는 형태와 같은 의미를 가진다. [2]

따라서 아래 명령어는 서버 로그를 `result.log` 파일에 계속 쌓아가며 백그라운드에서 서버를 실행하는 명령어라고 볼 수 있다.

```bash
nohup java -jar linux-springboot-0.0.1-SNAPSHOT.jar >> result.log 2>&1 &
```

## 7. 로그에서 포트 충돌 에러 확인하기

`result.log` 파일을 확인했을 때 아래와 같은 메시지가 보일 수 있다.

```text
Port 8080 was already in use
```

이 메시지는 8080 포트를 이미 다른 프로세스가 사용하고 있어서 Spring Boot 서버가 정상적으로 실행되지 못했다는 의미다.

즉, 백그라운드에서 실행한 프로세스는 화면에 로그가 바로 보이지 않기 때문에, 문제가 생겼을 때 로그 파일을 직접 확인해야 한다.
---

## 정리

`nohup`으로 프로그램을 실행하면 기본적으로 로그가 `nohup.out` 파일에 저장될 수 있다.

```bash
nohup java -jar linux-springboot-0.0.1-SNAPSHOT.jar &
cat nohup.out
```

원하는 파일에 로그를 남기고 싶다면 표준 출력과 표준 에러 출력을 직접 리다이렉션하면 된다.

```bash
nohup java -jar linux-springboot-0.0.1-SNAPSHOT.jar >> result.log 2>&1 &
cat result.log
```

백그라운드 프로세스는 터미널에 로그가 바로 보이지 않으므로, 문제가 생겼을 때는 `nohup.out` 또는 직접 지정한 로그 파일을 확인해야 한다.
